# 评分量规 (Scoring Rubric)

7 个审计维度，每维度 1-5 分。评分时**必须**考虑 skill 的实际用途和领域约束，不可机械套用。

---

## D0: 核心逻辑与 skill 适配性

**分析 skill 究竟在干什么，这条核心逻辑适不适合用 skill 来实现。**

### 分析路径

1. 一句话概括这个 skill 的核心逻辑：它让 AI 做什么？在什么场景下触发？
2. 判断这个逻辑是否适合 skill 机制：
   - **适合 skill**：需要领域知识注入、需要约束 AI 行为模式、需要标准化多步骤流程、需要让 AI 获得它训练数据中没有的专业知识
   - **不适合 skill**：AI 凭自身能力就能做好的简单任务、一次性脚本能解决的事、纯粹的信息查询（不需要行为约束）

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 核心逻辑非常适合 skill，离开 skill AI 基本无法完成任务 |
| 4 | 适合 skill，skill 显著提升了任务质量或效率 |
| 3 | 用 skill 有一定价值，但裸 AI 也能做得差不多 |
| 2 | skill 必要性存疑，核心逻辑更像是"把简单问题复杂化" |
| 1 | 完全不适合用 skill，应该用脚本/配置/其他方式解决 |

### 判断要点

- 核心逻辑是否利用了 skill 的核心优势（知识注入 + 行为约束）？
- 如果去掉这个 skill，AI 能否仅凭系统 prompt 达到同样效果？
- skill 是否在解决一个"真问题"，而非过度工程化？

## D1: 模块化与加载量

**分析 skill 的加载是否合理，内容组织是否支持按需加载。**

### 分析路径

1. 统计各文件大小和总大小
2. 分析文件之间的依赖关系
3. 判断是否存在"每次加载都读了但用不上"的内容

判断时区分"合理的大"和"不合理的大"：

- **合理的大**：核心逻辑确实复杂，拆分文件反而增加 AI 的理解成本；单个 reference 文件虽然大但在对应场景下全部内容都有用；skill 处理的领域知识本身就稠密（如数学公式、法律条文）
- **不合理的大**：多个互斥的配置/风格/模式塞在同一个文件里，每次加载只能用到其中一种；SKILL.md 中混入了大量本应属于 reference 的细节内容；存在写了但实际用不上的"僵尸内容"

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 模块化完美，SKILL.md 精炼，references 按需加载，无冗余 |
| 4 | 整体合理，有少量可优化的组织问题 |
| 3 | 存在一些不合理的大文件或缺少拆分，但不影响核心功能 |
| 2 | 有明显的模块化问题（如互斥内容未拆分），显著浪费 context |
| 1 | 大量冗余，AI 每次都要读一堆用不上的内容 |

### 判断要点

- 判断文件是否"过大"的唯一标准是：在需要它的场景下，其内容是否大部分都会被用到。回答"是"则合理，回答"否"则不合理
- 文件数量多不等于模块化好——如果 10 个文件每次都要全加载，不如合并为 1 个

## D2: 逻辑一致性

**分析 skill 中是否存在前后矛盾、模糊不清的指令，可能导致 AI 产生幻觉或行为不一致。**

### 分析路径

1. 检查是否存在**直接矛盾**：A 段说做 X，B 段说不要做 X
2. 检查是否存在**模糊指令**：用了"适当"、"合理"、"尽量"等词但没有给出判断标准
3. 检查是否存在**未定义术语**：用了内部概念但没有在 skill 中定义
4. 检查是否存在**不可达路径**：路由规则中写了某个分支，但触发条件永远不满足

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 指令明确一致，无矛盾，模糊处有判断标准 |
| 4 | 基本一致，个别措辞可更精确 |
| 3 | 存在一两处轻微矛盾或模糊，但 AI 大概率能正确理解 |
| 2 | 存在多处矛盾或关键指令模糊，AI 容易走偏 |
| 1 | 严重矛盾，AI 几乎不可能按预期执行 |

### 判断要点

- "简洁回答" vs "详细解释每个步骤" 同时出现但没有说明何时用哪种 → 矛盾
- "必须用 A 模板" 但后面给出的示例用了 B 格式 → 矛盾
- "根据用户水平调整" 但没有给出判断水平的信号 → 模糊

## D3: 简洁性与信息密度

**分析 skill 是否用精炼的语言描述了核心逻辑，避免无用功和"水篇幅"。**

### 分析路径

1. 识别 skill 中的"水内容"：
   - 同一件事反复说多遍
   - 大段的口语化铺垫/客套话（注意：如果 skill 本身是角色扮演类，口语化示例是**合理的**）
   - 可以通过系统 prompt 默认行为覆盖的冗余约束
2. 识别 skill 中的"缺失内容"：
   - 关键决策点没有说明
   - 异常处理路径缺失

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 信息密度极高，每句话都有存在的理由，无废话 |
| 4 | 整体精炼，少量可压缩的内容 |
| 3 | 有一些口语化或重复，但不太影响效率 |
| 2 | 明显的水内容，可压缩 30%+ 而不影响信息量 |
| 1 | 大量冗余，核心信息被稀释在无关内容中 |

### 判断要点

如果 skill 的**目的本身就是让 AI 模仿某种口吻/风格**（如角色扮演 skill），那么口语化示例、性格描写等等是**功能性的，不是水内容**。要区分"无意义的水"和"服务于 skill 目标的风格示范"。

## D4: 代码执行与环境配置

**分析 skill 是否包含脚本执行、环境配置、依赖管理等内容，以及这些内容的处理是否合理。**

### 分析路径

1. 是否有 scripts/ 目录？里面的脚本是做什么的？
2. SKILL.md 中是否有要求 AI 运行命令（pip install、conda install 等）？
3. 是否指定了依赖版本？版本约束是否合理（锁死 vs 宽松）？
4. 是否有帮助用户配置环境的说明？

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 脚本组织合理，环境配置清晰，版本约束恰当，有回退策略 |
| 4 | 有脚本/环境配置，基本合理，个别细节可优化 |
| 3 | 有相关内容但不够完善（如没指定版本、没检测环境） |
| 2 | 脚本逻辑有问题，或环境配置有明显缺陷 |
| 1 | 不该有脚本却硬塞了，或者脚本有安全风险 |
| N/A | skill 不需要执行代码，此项不适用 |

### 判断要点

- 脚本中硬编码了绝对路径 → 可移植性差
- `pip install` 没指定 `--quiet` 或没有错误处理 → 可能导致交互式阻塞
- 指定了 `package==1.2.3` 精确版本但没说明原因 → 应该用 `>=` 除非有兼容性理由
- 脚本没有 shebang 或执行权限说明 → 用户可能不知道如何运行

## D5: 大文件处理策略

**分析 skill 是否会面临处理大文件的风险，以及应对策略是否合理。**

### 分析路径

1. 识别 skill 中涉及文件处理的场景：
   - 读取用户提供的文件（PDF、CSV、代码仓库等）
   - 生成输出文件（报告、图表、数据文件等）
   - 扫描目录/搜索文件
2. 分析每种场景的**风险**：
   - 文件类型是否有大小上限风险（PDF 多页、CSV 百万行、代码仓库上千文件）
   - skill 是否给出了**分页/分块/限制**策略
   - 是否有超时或 token 爆炸的风险
3. 判断**合理性**：
   - 如果 skill 的设计目标就是处理大文件（如 PDF 提取），那它**必须**有分页/分段策略
   - 如果 skill 预期的文件都是小文件（如单页配置），那不需要过度设计
   - 如果 skill 完全没有考虑文件大小问题但场景涉及大文件 → 这是一个风险点

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 充分考虑了大文件场景，有明确的分页/分块/限制策略 |
| 4 | 基本覆盖了大文件风险，策略可行 |
| 3 | 意识到大文件问题但策略不够具体 |
| 2 | 明显涉及大文件场景但未做任何处理，有卡死风险 |
| 1 | 设计上有严重缺陷，大文件场景必然会出问题 |
| N/A | skill 不涉及文件处理，此项不适用 |

### 判断要点

- skill 的设计目标就是处理大文件但没有任何分页/分段策略 → 这是致命缺陷
- skill 预期的文件都是小文件（如单页配置）→ 不需要过度设计，不扣分
- skill 完全没有考虑文件大小问题但场景涉及大文件 → 风险点，扣分
- 策略是否具体可执行：光说"要注意大文件"不算策略，给出具体阈值才算

## D6: Skill 设计规范遵守

**分析 skill 是否遵守 skill 设计的基本规范。此项酌情评分，不过于严苛。**

### 分析路径

| 检查项 | 说明 |
|--------|------|
| SKILL.md 格式 | 是否有 YAML frontmatter（name, description） |
| description 质量 | 是否包含触发条件和使用场景（而非仅功能描述） |
| references 组织 | 是否合理使用 references/ 做渐进加载 |
| scripts 组织 | 是否合理使用 scripts/ 存放可执行代码 |
| 无冗余文件 | 目录下是否有多余的非规范文件 |
| 行数合理 | SKILL.md 是否控制在 500 行以内（酌情） |

### 评分标准

| 分 | 含义 |
|----|------|
| 5 | 完全遵守规范，目录结构清晰，description 触发条件明确 |
| 4 | 基本遵守，有一些小的不规范但不影响使用 |
| 3 | 部分遵守，有明显但不致命的规范问题 |
| 2 | 多项不遵守，影响 skill 的可维护性和触发准确性 |
| 1 | 几乎不遵守规范，或目录结构混乱 |

### 判断要点

- 规范是为功能服务的，不要本末倒置。SKILL.md 超过 500 行但内容都是必需的 → 不扣分
- description 质量是实际影响最大的检查项——它直接决定 skill 是否能被正确触发
- 缺少 references 目录但 SKILL.md 本身就足够精简 → 不扣分

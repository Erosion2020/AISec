# 架构评审视角

用于专家级代码评审时的检查清单和分析角度。不是每段代码都需要全部覆盖——根据用户的问题选择相关角度。

---

## 正确性

- **维度匹配**：`nn.Linear(in, out)` 的上层输出和下层输入是否对齐？squeeze/unsqueeze 是否正确？
- **设备一致性**：模型和数据是否都在同一设备上？有没有漏 `.to(device)` 的地方？
- **train/eval 模式**：训练前是否调了 `model.train()`？验证前是否调了 `model.eval()`？`torch.no_grad()` 是否包住了验证/推理代码？
- **梯度管理**：`optimizer.zero_grad()` 是否在每次迭代的开头？验证/推理代码是否被 `torch.no_grad()` 包裹？
- **数据拆分**：验证集和测试集是否与训练集独立？有没有数据泄漏（如先标准化再拆分）？
- **标签处理**：回归任务的标签是否需要 squeeze？分类任务的标签是否正确编码？

---

## 可读性

- **命名**：变量名是否反映其含义？`x_train` > `X`，`mean_valid_loss` > `loss2`
- **魔法数字**：`0.2`、`256`、`1e-5` 等硬编码值是否应该提取到 config 字典中？
- **注释质量**：注释是在解释"为什么"还是在重复代码？`# 将梯度置0` 是好的（解释了用意），`# 调用 zero_grad` 是坏的（重复代码）
- **代码结构**：是否有关注点分离？数据处理、模型定义、训练逻辑是否各自独立？

---

## 性能

### 数据加载
- `DataLoader` 的 `num_workers` 是否设置？（默认 0 表示主进程加载，大 batch 时会成为瓶颈）
- `pin_memory=True` 是否开启？（加速 CPU→GPU 数据传输）
- 数据预处理是在 `Dataset.__getitem__` 里做还是在之前做？预处理一次 > 每个 epoch 重复做

### 模型训练
- 不必要的 `.detach()` / `.cpu()` 调用——如果后续还要训练，中途 detach 会断掉计算图
- 验证过程中是否关闭了梯度计算？
- `with torch.no_grad()` 是否覆盖了整个验证循环？

### 内存
- 不再使用的 DataFrame/大数组是否 `del` 了？（尤其对大文件有意义）
- 中间变量是否及时释放？如在循环中持续 `append` 但不在后续使用

---

## ML 专项

### 训练流程合理性
- **早停 patience**：`early_stop=400` 配合 `n_epochs=3000` 是否合理？loss 下降趋势如何？
- **学习率**：`1e-5` 对 SGD 来说可能偏小——是否有 warmup 或学习率衰减策略？
- **验证集比例**：`valid_ratio=0.2` 对当前数据量是否足够？
- **模型复杂度**：`input→16→8→1` 对 118 维特征是否太简单或太复杂？
- **Batch Size**：`256` 对 2699 条数据只有 ~10 个 batch/epoch，梯度更新频率是否足够？

### 评估完整性
- 是否有验证集？还是只在测试集上评估？
- 回归任务用 MSE 可以，但如果有时间序列特征（这里是 COVID 数据），是否应该考虑时序切分而非随机切分？

### 复现性
- 是否设置了所有随机种子（NumPy、PyTorch、cuDNN）？
- 数据处理是否有随机性？

## 评审输出格式

当进行架构评审时，输出按此结构：

```
## 架构评审：<文件/函数名>

### 整体评价
<一句话总结代码质量，指出最大的优点和最大的问题>

### 正确性
- <发现的问题或不确认的点>

### 可读性
- <建议>

### 性能
- <建议>

### 改进建议
<如果有值得改进的地方，给出具体的代码对比——不是泛泛地说"可以优化"，而是展示改写前后的代码>
```
